## 四、当前功能缺陷

### 4.1 P0：匿名提问箱前端路径错误

**类别：前后端接口不一致**  
**组别：插件系统 / 匿名提问箱**

**现象：**

前端 `questionBoxApi` 调用：

```text
/plugins/question-box/questions/
/plugins/question-box/questions/inbox/
/plugins/question-box/questions/{id}/answer/
/plugins/question-box/questions/{id}/delete/
```

后端实际提供：

```text
/plugins/question-box/
/plugins/question-box/inbox/
/plugins/question-box/{id}/answer/
/plugins/question-box/{id}/delete/
```

**影响：**

匿名提问箱页面的提问、收件箱、回答、删除都会打到不存在路径，功能不可用。

**代码依据：**

- `frontend/src/api/plugin.js`
- `backend/apps/plugins/questions/urls.py`
- `frontend/src/views/QuestionBoxPage.vue`

**建议：**

优先改前端 API 封装，去掉多余的 `questions/` 层级。若希望后端路径更符合文档，也可以改后端路由，但要同步所有调用点。

### 4.2 P0：小组创建前端路径错误

**类别：前后端接口不一致**  
**组别：兴趣小组**

**现象：**

前端：

```text
POST /api/v1/groups/
```

后端：

```text
POST /api/v1/groups/create/
```

**影响：**

`GroupListPage.vue` 中创建小组会失败。

**代码依据：**

- `frontend/src/api/plugin.js`
- `frontend/src/views/GroupListPage.vue`
- `backend/apps/groups/urls.py`

**建议：**

短期改 `groupApi.create` 为 `/groups/create/`。长期应决定是否统一 REST 风格，若统一则调整后端路由。

### 4.3 P0：WebSocket 实时收消息事件处理错误

**类别：实时通信协议不一致**  
**组别：聊天系统**

**现象：**

后端下发：

```json
{
  "type": "new_message",
  "data": { ... }
}
```

前端处理：

```javascript
if (data.type === 'message' && data.room_id) {
  ...
  this.messages[rid].push(data.message)
}
```

**影响：**

REST 发送自己的消息可以显示，但 WebSocket 收到其他人的实时消息不会写入消息列表。聊天实时性验收会失败。

**代码依据：**

- `backend/apps/chat/consumers.py`
- `frontend/src/stores/chat.js`

**建议：**

前端改为监听 `new_message`，从 `payload.data` 读取消息，并按 `room_id` 归入 `messages`。

### 4.4 P1：首页发布动态话题字段传错

**类别：字段契约不一致**  
**组别：社交广场 / 话题**

**现象：**

前端发布动态提交：

```javascript
formData.append('tag', newTag.value)
```

后端读取：

```python
topic_tag = serializer.validated_data.get("topic_tag", "")
```

**影响：**

用户发布动态时选择的话题不会保存。后续按 `topic` 筛选时，新发布动态可能无法被筛出。

**代码依据：**

- `frontend/src/views/HomePage.vue`
- `backend/apps/feed/serializers.py`
- `backend/apps/feed/views.py`

**建议：**

前端字段改为 `topic_tag`。

### 4.5 P1：小组动态查询混入成员个人动态

**类别：业务逻辑偏差**  
**组别：兴趣小组 / 动态隔离**

**现象：**

`GroupPostsView` 当前查询：

```python
Post.objects.filter(
    Q(group_id=pk) | Q(author__in=User.objects.filter(groupmember__group=group))
)
```

这意味着小组成员所有个人动态都会出现在小组动态页。

**影响：**

违反“组内动态隔离”的产品预期。非组内动态可能被错误展示到小组页，造成隐私和内容边界问题。

**代码依据：**

- `backend/apps/groups/views.py`

**建议：**

组内动态只应查询 `Post.objects.filter(group_id=pk)`。如果要显示成员公开动态，应作为另一个明确模块，而不是“组内动态”。

### 4.6 P1：发动态文件校验顺序可能产生半成功数据

**类别：数据一致性风险**  
**组别：社交广场 / 发布动态**

**现象：**

`PostCreateView` 先创建 Post，再校验文件附件类型和大小。若后续某个文件非法，接口返回错误，但 Post 和已保存图片可能已经落库。

**影响：**

用户看到“发布失败”，数据库却产生了动态或部分附件，造成脏数据。

**代码依据：**

- `backend/apps/feed/views.py`

**建议：**

在创建 Post 前完成所有图片和文件校验，或使用 `transaction.atomic()` 包裹整个创建过程。

### 4.7 P1：聊天未读数逻辑不完整

**类别：状态同步缺陷**  
**组别：聊天系统 / 未读计数**

**现象：**

`last_read_at` 为空时使用 `timezone.now()`，导致没有读时间的房间默认未读为 0。同时没有看到进入聊天室后更新 `last_read_at` 的逻辑。

**影响：**

未读数可能不准确。新加入房间或历史消息场景尤其明显。

**代码依据：**

- `backend/apps/chat/serializers.py`
- `backend/apps/chat/views.py`
- `frontend/src/views/ChatRoomPage.vue`

**建议：**

明确未读规则：进入聊天室加载消息后更新当前成员 `last_read_at`。`last_read_at` 为空时应按业务选择“全部历史算未读”或“加入前消息不算未读”，不能默认 `now()` 后永远 0。

### 4.8 P2：默契问答匹配度分母可能不符合预期

**类别：产品算法偏差**  
**组别：插件系统 / 默契问答**

**现象：**

`_update_result()` 中 `total = questions.count()`，包括双方之间所有题目，不只已回答题。

**影响：**

未回答题会拉低匹配度。PRD 描述更接近“答题后按已回答题计算匹配度”。

**代码依据：**

- `backend/apps/plugins/quizzes/views.py`

**建议：**

产品上明确分母：若按已回答题，分母应为 `QuizAnswer` 数量；若按总出题数，需要在 UI 上解释“未回答视为错误”。

### 4.9 P2：默契问答后端缺少业务校验

**类别：权限/业务规则缺口**  
**组别：插件系统 / 默契问答**

**现象：**

`QuizQuestionCreateSerializer` 只校验字段存在，没有看到以下校验：

- 目标用户是否存在
- 是否允许给自己出题
- 是否必须是好友

前端通过好友列表选择目标，但后端 API 可被绕过。

**影响：**

如果 PRD 坚持“向好友出题”，当前后端规则不完整。

**代码依据：**

- `backend/apps/plugins/quizzes/serializers.py`
- `backend/apps/plugins/quizzes/views.py`

**建议：**

在 serializer 或 view 中补目标用户存在、自身排除、好友关系校验。

### 4.10 P2：后台日志序列化对空 user 不稳

**类别：健壮性风险**  
**组别：管理后台 / 操作日志**

**现象：**

`SiteLog.user` 允许为空，但 `AdminLogSerializer` 直接使用 `source="user.nickname"`。

**影响：**

若存在系统日志或历史日志 `user=null`，日志接口可能序列化失败。

**代码依据：**

- `backend/apps/admin_dashboard/models.py`
- `backend/apps/admin_dashboard/serializers.py`

**建议：**

改为 `SerializerMethodField`，空用户显示“系统”或 `null`。

### 5.1 匿名提问是“对普通用户匿名”，不是“完全匿名”

**类别：产品定义风险**  
**组别：匿名提问箱**

代码中 `AnonymousQuestion.sender` 是必填外键，管理员能追溯真实提问者。这符合 PRD 的“管理员可见真实提问者”，但不符合“数据库层完全匿名”。

**建议：**

产品文案统一为：

> 对普通用户匿名，管理员在必要情况下可追溯。

避免承诺“完全匿名”。

### 5.2 插件系统仍未完全平台化

**类别：架构技术债**  
**组别：插件系统**

后端插件注册机制较完整，但前端路由仍硬编码：

- `/profile/:userId/plugins/quiz`
- `/profile/:userId/plugins/question-box`

当前只有两个固定插件时可接受。若后续频繁新增插件，需要实现动态路由和插件页面注册。

### 5.3 前端 Store 不完整但当前可运行

**类别：状态管理技术债**  
**组别：前端架构**

代码中只有：

- `auth`
- `feed`
- `chat`
- `friend`

没有 `notificationStore` 和 `pluginStore`。当前页面直接调用 API，不阻塞短期功能，但会影响跨页面通知角标、插件状态同步和复杂交互。

### 5.4 生产部署仍不是当前版本能力

**类别：交付边界**  
**组别：部署运维**

代码和文档均显示当前以开发环境为主：

- Docker Compose 只有 PostgreSQL + Redis
- 无应用 Dockerfile
- 无 Nginx
- 默认入口指向 dev settings

这不阻塞本地演示，但如果目标是公网部署，需要单独立项。
